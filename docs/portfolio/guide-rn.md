# Гайд для написания Release Notes: понятно,结构化, без паники

**Цель документа:** Помочь разработчикам, тестировщикам и техписателям создавать Release Notes, которые пользователи будут *читать*, а не *закрывать*.

## 1. Зачем нужны хорошие Release Notes?

- **Уменьшают хаос:** Пользователь видит, что поменялось, до того, как сломалась его задача.
- **Снижают нагрузку на поддержку:** Вместо 100 тикетов «А где эта кнопка?» — 10.
- **Формируют доверие:** Регулярные и понятные заметки = профессиональная команда.

## 2. Структура идеального Release Notes (шаблон)

Ниже — проверенная структура. Скопируйте её и заполняйте для каждого релиза.

### Заголовок: [Версия] — [Краткая суть] — [Дата]

*Пример:* `v2.4.0 — Новый дашборд и исправление экспорта — 2025-03-15`

### 2.1. 🔖 О релизе (TL;DR)
*Одним абзацем для занятого менеджера: что главного произошло.*

> **Пример:** В этой версии мы полностью переработали экран аналитики, ускорили загрузку отчётов и исправили критическую ошибку при выгрузке в Excel.

### 2.2. 🆕 Новые возможности (New Features)
*Для каждой фичи — ссылка на документацию (если есть).*

- **Модуль прогнозирования продаж** (PRO-1234)
    - Теперь ИИ предсказывает выручку на следующую неделю.
    - *Где найти:* Дашборд → вкладка «Прогноз».
    - *Инструкция:* [Ссылка на гайд]
- **Тёмная тема** (UI-567)
    - Включите в настройках профиля, глаза скажут спасибо.

### 2.3. ⚙️ Улучшения (Improvements)
*Что стало быстрее, удобнее, красивее. Не про новый функционал, а про апгрейд старого.*

- **Поиск клиентов** (PERF-89) — скорость увеличена на 40% даже при 10k+ записях.
- **Массовый импорт** (DATA-22) — теперь можно загрузить CSV до 50 МБ (было 10 МБ).

### 2.4. 🐛 Исправления (Bug Fixes)
*Что починили. Хороший тон — указывать тикеты баг-трекера.*

- Исправлена ошибка, из-за которой дублировались счета при повторной отправке (BUG-445).
- Больше нельзя сохранить заказ с отрицательным количеством товаров (BUG-789).
- **Важно:** Пофикшен редкий краш при открытии PDF-отчёта на macOS Sonoma.

### 2.5. ⚠️ Изменения, ломающие обратную совместимость (Breaking Changes)
*Красная зона. Пишите максимально понятно, даже если кажется, что «и так очевидно».*

> **🚨 Внимание:**
> - Устаревший эндпоинт `/api/v1/orders` будет удалён через 2 недели. Используйте `/api/v2/orders`.
> - Поле `user.status` изменено с текстового ('active') на булево (`is_active: true`).

### 2.6. 📦 Известные проблемы (Known Issues)
*Честно, но осторожно. Чтобы пользователь не искал то, чего нет.*

- На медленных соединениях аватарка может грузиться до 3 секунд (исправим в v2.4.1).
- В мобильной версии не скроллится длинный чек-лист (обход: перевернуть телефон).

### 2.7. 📅 Как обновиться? (Update Instructions)
- **SaaS:** Обновление произошло автоматически. Просто перезагрузите страницу (Ctrl+F5).
- **On-premise:** Скачать новый билд можно в [личном кабинете]. Команда: `docker pull ...`

## 3. Плохо vs Хорошо (примеры)

| Плохо (бесполезно) | Хорошо (понятно) |
|-------------------|------------------|
| "Проведена оптимизация." | "Дашборд загружается на 60% быстрее за счёт кэширования." |
| "Исправлены мелкие баги." | "Исправлен баг, из-за которого кнопка 'Удалить' работала без подтверждения." |
| "Улучшен интерфейс." | "Добавлены подсказки при наведении на иконки отчётов." |

## 4. Чек-лист перед публикацией

- [ ] Название версии соответствует стандартам (SemVer / дата).
- [ ] Нет фраз «разные исправления» и «улучшена производительность» без деталей.
- [ ] Для каждого *Breaking Change* есть пример «было → стало».
- [ ] Release Notes понятны новому сотруднику без знания внутреннего контекста.
- [ ] Ссылки на полную документацию работают.
- [ ] Дата указана (или версия нумерована).
